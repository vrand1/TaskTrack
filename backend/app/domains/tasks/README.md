# Домен задач

Этот слой отвечает за основной бизнес-поток сервиса: жизненный цикл задачи, связи с проектом, назначение исполнителя и доменные события.

## Структура каталогов

```
api/            # router, tag_router, dependencies
services/       # TaskService
repositories/   # задачи, комментарии, справочники
params/         # параметры списков и фильтров
schemas/        # Pydantic DTO
domain/         # FSM, refs, history, constants
models/         # ORM
```

## Ключевые решения

- **FSM статусов** (`todo -> in_progress -> review -> done`) вынесен в отдельную логику, чтобы правила переходов были централизованы и тестируемы.
- **Статусы и приоритеты:** справочник в БД (`task_statuses`, `task_priorities`) — задел под настраиваемый workflow (порядок FSM по `sort_order`, реестр при старте). Сейчас новый код в БД без обновления constants не попадёт в REST.
- **Reopen (`done -> todo`)** — отдельный эндпоинт `POST /tasks/{id}/reopen`, не проходит через FSM `PATCH /status`; допускается один раз, флаг `was_reopened` блокирует повтор.
- **Подзадачи** поддерживаются через `parent_task_id` с валидацией консистентности (один проект, запрет циклов).
- **История изменений** фиксирует значимые действия (создание, обновление, смена статуса, комментарии, удаление/восстановление).

## Точки роста

- Вынести публикацию доменных событий в брокер (Kafka/RabbitMQ) без изменения API слоя.
- Добавить SLA/эскалации (просрочки, авто-уведомления) как отдельные policy-обработчики.
- Разделить read/write-модели (CQRS), если вырастет нагрузка на списки и аналитику.
