# Тестовое Задание - Python Backend

**Срок сдачи:** 3-5 дней  
**Минимум в ТЗ:** Python ≥3.11, FastAPI, Docker

---

## Контекст

Ты делаешь внутренний сервис для команды разработки. Команда создаёт задачи вручную, назначает их на себя и двигает по статусам. Инфраструктура и способ сдачи должны быть близки к тому, как обычно ведутся бэкенд-проекты.

Мы смотрим решение как на мини-прод-сервис.

---

## Что Нужно Реализовать

Сервис должен позволять:

- создать задачу с названием, описанием и исполнителем;
- получить список задач с фильтрацией по статусу и исполнителю;
- получить одну задачу по идентификатору;
- изменить поля задачи;
- удалить задачу (soft-delete или hard-delete - опиши выбор в README).

**Статусы:** `todo` -> `in_progress` -> `review` -> `done`. Переход назад из `done` невозможен - верни понятную ошибку.

Дополнительные статусы или правила переходов - по желанию, с обоснованием в README.

Поля модели можно расширять, если это улучшает сервис - опиши в README, зачем.

---

## Креатив

Сверх списка поощряются осмысленные решения, например:
- пагинация, сортировка, поиск по названию;
- поля: приоритет, дедлайн, теги, `created_at` / `updated_at`;
- история смены статусов, комментарии к задаче;
- единый формат ошибок с кодами;
- идемпотентность там, где уместно.

В README - отдельный блок **Дополнительные фичи**: что добавил в API и модели, зачем это команде.

Хорошо описанные фичи сверх минимума будут плюсом, если обязательные требования закрыты.

---

## Стек

**Python + FastAPI + Docker**. Остальное - **как в типичном прод-проекте**: PostgreSQL, миграции, `docker compose`.

Стек в ТЗ описан коротко намеренно: важно, **как ты сам соберёшь инфраструктуру** под сервис.

---

## Требования
### Обязательно

- Python ≥3.11, FastAPI.
- Авторизация на JWT или простых токенах.
- `docker compose`.
- Миграции.
- Валидация входных данных и понятные сообщения об ошибках для клиента.
- README: как запустить, стек, закрытие минимума ТЗ, блок **Дополнительные фичи**.
- Кэширование списка задач в **Redis** - с обоснованием TTL.
- Тесты на создание задачи и смену статуса (другие сценарии приветствуются).
- CI (GitHub Actions/GitLab CI): линтер и тесты.
